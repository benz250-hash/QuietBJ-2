QuietBJ 缓存导出 / 导入说明（v513）

一、缓存文件用途
- 文件名：community_building_cache.json
- 作用：保存“用户查询后识别出的楼栋点”
- 用途：给遮挡算法提供同小区楼栋点云，判断“前楼是否挡住后楼”

二、为什么需要导出 / 导入
- 部署环境重启后，本地可写文件可能丢失
- 所以需要把缓存导出到本地 JSON 备份
- 重启后再导入恢复

三、导出文件格式
顶层结构：
- schema_version：缓存导出格式版本
- exported_at：导出时间（UTC）
- app_version：导出时的工程版本
- stats：统计信息
  - community_count：小区数
  - building_count：楼栋数
- communities：真正的小区缓存数据

四、communities 里的字段
- 小区名：例如“国奥村西区”
  - source：来源（manual / query_trace / import_merge）
  - updated_at：更新时间
  - buildings：楼栋列表

每栋楼的字段：
- name：完整显示名，例如“国奥村西区16号楼”
- building_token：楼号短标识，例如“16号楼”“a13号楼”
- lon：经度
- lat：纬度

五、导入方式
1. 合并导入
- 保留当前缓存
- 用“community_name + building_token”合并导入文件中的楼栋
- 已存在的楼栋会更新坐标与名称
- 不会删掉当前缓存里其他楼栋

2. 完全覆盖
- 直接用导入文件替换当前缓存
- 适合你确认本地备份才是“真版本”的情况

六、规模与性能
- 这个缓存按“小区”分组，不是全国数据库
- 每栋楼只存 name / building_token / lon / lat 四类核心信息
- 一般几百到几千栋楼，文件体积仍然在可控范围
- 遮挡算法主要在“同小区楼栋点”范围内计算
